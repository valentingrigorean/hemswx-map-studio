# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

HemsWX Map Studio is a browser-based workbench for the mobile app's remote configuration. It edits the `AppBootstrapConfig` envelope served by `config.dev.hemswx.no` / `config.hemswx.no` — most importantly its `map_layers` field (the v2 map-layer schema parsed by `lib/features/remote_config/` in the Flutter app) — plus Portal-hosted basemaps and `RemoteAppConfiguration` templates.

Stack: Preact + @preact/signals, TypeScript, Vite, Tailwind (v4, CSS-based config), ArcGIS Maps SDK for JavaScript, Monaco editor.

## Shell layout

`App.tsx` renders a labeled rail (`components/studio/Rail.tsx`) that switches between sections; the active section is `activeNav` in `lib/studio.ts`. The dev/prod environment switch lives in the rail's Environment card (not the top bar) and scopes all persisted documents per env (`envKey`/`readScoped` in `lib/studio.ts`, switch logic in `lib/workspace.ts`).

| Section | Component | State |
|---|---|---|
| Layers | `components/editor/EditorApp.tsx` (Navigator, Inspector, SublayerCard, ArcGIS preview) | `lib/v2/store.ts` — `MapConfigV2`, the source of truth |
| Basemaps | `components/basemaps/PortalBasemaps.tsx` — live Portal group, tag write-back | `lib/basemaps/store.ts` + `lib/basemaps/portal.ts` |
| Catalog | `components/catalog/CatalogPanel.tsx` — built-ins ⊕ remote union | `lib/catalog/` (`builtins.generated.json`) |
| Templates | `components/templates/` — `RemoteAppConfiguration` editor; viewpoint is picked on a map (`ViewpointPicker.tsx`, drag-rectangle via SketchViewModel, presets + rotation). The picker writes the ArcGIS envelope form `{xmin,ymin,xmax,ymax}` (`lib/template/viewpoint.ts`) because the app's GeoEnvelope `toJson` (`{bbox:[…]}`) does not parse back through `parseGeoJson`. The editor's Preview button jumps to the Preview section | `lib/template/store.ts` |
| Preview | `components/preview/` — the mobile Map screen recreated with the real ArcGIS SDK: full-bleed map (2D `arcgis-map` / 3D `arcgis-scene` per `dimension`) composing the loaded Layers doc ⊕ an applied configuration ⊕ a basemap, plus click-to-identify and the POI toolbar. The floating panel (`ControlPanel.tsx`) has three tabs mirroring the app's Map-screen functions: **Layers** (`LayersTab.tsx`, the main one) renders the document's feature tree with the app's select semantics — state is item-keyed `mapFeatures{itemId}` exactly like `LayerKey.custom(item.id)` (a leaf is its own single item, `UIFeatureEntity.fromJson` parity via `featureTreeOf`/`liveFeatures`), `select: one` enables an item and explicitly disables siblings (`_toggleItem` in `dynamic_feature_section.dart`), `select: many` toggles independently; enabled items draw their sublayers. **Configs** lists `RemoteAppConfiguration`s and applies with mergeStrategy semantics (`lib/preview/options.ts` mirrors `applyTo` in `app_configuration_extensions.dart`; top-of-map chips do the same). **Editor** (secondary) edits the rest of the working MapOptions blob — basemap flyout, 2D/3D, built-in toggles (listed, not drawn), POI, diff chips — and owns the apply bar; nothing is written until **Apply to template** (or **Save as new**, `addTemplateFrom`). The legacy `edit` flag is gone — the app never reads it, so `normalize` strips it on ingest and nothing gates on it. The basemap is preview-only by default: a basemap FAB on the map switches it ad hoc, and `selectedBasemapId` is stripped from saves unless the Editor's **Save basemap in template** check is on (`pinBasemap`, auto-checked when the applied template pins one); the basemap diff chip compares would-be-saved vs template, not working vs baseline | `lib/preview/state.ts` — module-level working blob (survives section switches; a template seeds it only on explicit apply: a chip tap or the Templates Preview button, never on mount; effects prune `mapFeatures` ids missing from the current document and unlink a deleted applied config) |
| Tools | `components/tools/ToolsPanel.tsx` — migrate, validate, diff, format | `lib/tools/validate.ts`, `lib/v2/migrate.ts` |
| Settings | `components/SettingsPanel.tsx` | `lib/settings.ts`, `lib/credentials.ts` |

## Data flow

- **Load from URL** → `lib/importUrl.ts` + `components/studio/ImportUrlDialog.tsx`: fetches an API URL (hallo/bootstrap → `ingestRaw`; `hwx_template.php` → templates store) through the Vite `/wms-proxy` (the API has no CORS headers, so this needs the local dev/preview server). Known dev/prod sibling hosts (`wstest.snla-it.no` ↔ `ws.snla-it.no`, `config.dev.hemswx.no` ↔ `config.hemswx.no`) automatically import BOTH envs from the one URL by swapping the hostname (finishing on the pasted URL's env); a dev/prod target choice is shown only for unknown hosts. A hallo link additionally derives the templates endpoint from its apikey/signatur/id (`deriveTemplateUrl`): the filename comes from the fetched envelope's `api_info.templates` when present, defaulting to `hwx_templates.php` — the endpoint the app consumes (`ApiEndpoints.templates`); beware the legacy singular `hwx_template.php`, which serves a *different* document. One paste fans out to up to 4 fetches (hallo × dev/prod in parallel, then templates × dev/prod in parallel); each item is applied independently so one bad endpoint never blocks the rest. Failures are classified `soft` vs hard (`ImportResult.soft`): the derived templates endpoint and the auto-swapped sibling env are best-effort — a soft failure leaves that doc empty (`clearTemplates` for templates) and the dialog still auto-closes; only a hard error (the pasted env's own primary doc) keeps the dialog open. This matters because TEST's `api_info.templates` currently names the broken legacy singular `hwx_template.php` (DB2 error `Column FAVORITE not in table HWX_TEMPLATE_USERS`), so a dev+prod hallo import lands 3 docs and leaves dev templates empty rather than failing the whole import. The URL (it carries apikey/signatur) persists per-env in localStorage only. Each import records provenance per document (`lib/importSource.ts`: layers vs templates → `{url, kind, fetchedAt}`); `components/studio/SourceChip.tsx` shows it in the top bar with a Resync action (`resync()` re-fetches the same URL into the current env). Loading from file/paste/empty/clone clears the provenance so Resync can't overwrite non-URL documents.
- **Open/paste/drop** → `lib/v2/ingest.ts` `ingestRaw`: `splitDocument` (envelope store keeps every key except `map_layers`) → `coerceToV2` (auto-converts legacy v1 via `lib/v2/migrate.ts`) → `setConfig`. A v1 mirror (`lib/jsonStore.ts`) is kept only for the map preview's custom basemaps.
- **Baseline / dirty tracking** → `lib/baseline.ts`: every *load* (URL import, file, paste, resync, clone) snapshots the document per doc (layers/templates) and per env; edits never touch it (the templates JSON tab edits via `editTemplatesDoc`, not `loadTemplatesRaw`, so provenance + baseline survive). When current ≠ baseline a floating bar (`components/studio/DirtyBar.tsx`) offers Revert / Review changes / Download. **Review** (`lib/review.ts` + `ReviewDialog.tsx`) diffs baseline vs current (changed/added/removed, field drill-down); since there is **no deploy-to-server** the primary action is "Download & mark synced" — downloading the JSON is the deliverable, "mark synced" promotes the edits to the new baseline.
- **Re-sync** (refresh icon on the SourceChip) → `lib/reconcile.ts`: re-fetches the source URL; a clean document is replaced outright, but with local edits pending it runs a **3-way merge** (base = baseline, mine = current, theirs = fresh server copy — the server has no version/etag, items are keyed only by id). Only both-sides-changed collisions need a keep-mine/take-server pick (`ReconcileDialog.tsx`); after Apply, data = merge and baseline = theirs, so exactly the kept edits stay pending.
- **Download** → `lib/downloads.ts`; layers via `composeDocument(mapConfig.value)` re-wraps the envelope losslessly.
- **Compare dev ↔ prod** → `lib/compare.ts` + `components/studio/CompareDialog.tsx` (Compare button on the Layers and Templates top bars): diffs the two envs' working documents — the current env from the in-memory signals (fresher than the debounced persist), the other env from its scoped localStorage keys (`readScopedFor`). Layers diff at feature granularity (`weather/features/overrides` keyed `section:id`), templates keyed by id with the wrapped `AppConfiguration` flattened to `json.*` rows in the field drill-down. Comparison uses key-sorted stringify so import-order differences don't read as "Differs". The header shows each side's import provenance host (`sourceFor`), falling back to the env domain.
- **v2 schema** (`lib/v2/types.ts`): `{ schemaVersion: 2, weather[], features[], overrides[], baseMaps[]? }` with inline sublayers — no layer registry, no `intl`. Layer-type metadata and per-type validation live in `lib/v2/core.ts` (`LAYER_TYPES`, `sublayerIssues`).
- **Catalog** mirrors the Dart side: `scripts/gen-catalog.mjs` extracts `MapLayerId`, `LayerKey`, recognized override ids, etc. from the Flutter sources into `src/lib/catalog/builtins.generated.json` — never hand-edit it; run `npm run gen:catalog` after changing the Dart enums (`npm run build` fails if it drifts).

## Commands

```bash
npm run dev          # Vite dev server
npm run build        # checks generated catalog, then vite build
npm run gen:catalog  # regenerate builtins.generated.json from the Flutter sources
npm run deploy       # build + publish dist/ to gh-pages
```

## Conventions

- Shared UI primitives: `components/editor/primitives.tsx` (Field, TextInput, Segmented, StatusPill, TypeChip, ProvBadge, MapThumb) and `components/studio/primitives.tsx` (Btn, Pill, Toggle). Reuse these instead of inlining styled buttons/inputs.
- State is module-level @preact/signals stores under `lib/`; components read signals directly. Persistence is per-env localStorage via `envKey`.
- No comments in code. The docs above and this file carry the why; compiler/lint directives (`/// <reference>`, `@ts-expect-error`, `eslint-disable`) are the only exception.
- Dart model compatibility is the contract: field names, defaults, and per-type requirements must match the Flutter parser in `lib/features/remote_config/domain/`.
